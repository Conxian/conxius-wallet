# Community Funding Model for System Delivery (CON-137)

## Context
Internal funds are constrained. Community participation must help fund and execute suitable work while protecting internal-only deployment, wallet, and security-critical tasks.

## Funding Direction
- All bounty funds must come from the ALEX launch pathway once enabled.
- Until ConxianCSF is fully deployed via ALEX on Stacks, bounty work is claim-screened and approval-gated rather than payout-guaranteed.
- All bounty management aligns with ConxianCSF once fully deployed.

## Eligibility Matrix
| Issue Type | Category | Funding Source |
| :--- | :--- | :--- |
| Docs & Specs | Community | ALEX / ConxianCSF |
| UI/UX (Public) | Community | ALEX / ConxianCSF |
| Test Harnesses | Community | ALEX / ConxianCSF |
| Integration Adapters | Community | ALEX / ConxianCSF |
| Security-Critical | Internal-Only | N/A |
| Deployment/Signer | Internal-Only | N/A |
| Treasury/Principal | Internal-Only | N/A |

## Payout Readiness Checklist
1. PR merged and acceptance criteria met.
2. No sensitive material exposed (ZSE).
3. Bounty marked funded in ConxianCSF.
4. Maintainer confirmation of deployment parity.
